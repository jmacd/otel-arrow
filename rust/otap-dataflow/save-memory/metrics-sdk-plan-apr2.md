# Counter-Only OTAP Metrics SDK — Implementation Plan

## Problem

The otap-dataflow metrics instrumentation currently bridges through the OTel
SDK via `MetricsDispatcher`. We want a new codegen-driven OTAP metrics SDK
that encodes counters directly into OTAP Arrow record batches. The schema
is defined in OTel SemConv YAML (Weaver-compatible) and drives all code
generation.

**Scope**: Counters only (no histograms, gauges, updowncounters). Univariate
OTAP encoding. Level-aware dimensioning (Basic → fewer dimensions, Normal →
more dimensions).

## Key Design Insight

For each metric set, the OTAP encoding produces 3 tables:
1. **Metrics table** — one row per counter metric (name, type, unit,
   temporality, monotonic). **Precomputed at init time** (const/static).
2. **Attributes table** — one row per (data-point, attribute-key) pair for
   dimension attributes. **Precomputed at init time** (const/static).
3. **NumberDataPoints table** — one row per data point. **Built at runtime**
   from counter values during collection.

The metrics and attributes tables are fully determined by the schema and
the active MetricLevel, so they can be constructed once at startup. Only
the data points table needs to be built per collection tick.

## Approach

- **Weaver-first**: Use the OTel Weaver **Rust crates** (`weaver_semconv`,
  `weaver_resolver`, `weaver_resolved_schema`, `weaver_forge`) already in
  the workspace (pinned at `v0.21.2` from the `open-telemetry/weaver` git
  repo) to parse SemConv YAML and render MiniJinja templates. The existing
  codebase already uses these for schema resolution (in `crates/validation/`
  and `crates/core-nodes/fake_data_generator/`). We follow the same pattern:
  `RegistryRepo::try_new()` → `SchemaResolver::load_semconv_repository()` →
  `SchemaResolver::resolve()` → `ResolvedRegistry::try_from_resolved_registry()`
  → `weaver_forge` template rendering.
- **New crate `crates/metrics-sdk/`**: All runtime library types live here
  (Dimension trait, precomputed batch types, counter builders). This avoids
  modifying existing crate source directories.
- **xtask orchestration**: `cargo xtask generate-metrics` uses the Weaver
  Rust API directly (no subprocess). Add `weaver_*` crates to
  `xtask/Cargo.toml` as workspace dependencies.
- **MiniJinja templates**: Templates live in `templates/metrics/` in the
  workspace root. Rendered via `weaver_forge`.
- **Generated code checked in**: Generated `.rs` files are committed to
  the repo, like proto-generated code.
- **Pilot schema**: Model the consumer/producer counters
  (`node.consumer.items` / `node.producer.items`) with `outcome` dimension
  at Basic level, and `outcome` × `signal_type` at Normal level. Generated
  into `crates/telemetry/src/self_metrics/`.
- **Weaver dependency**: xtask gains `weaver_common`, `weaver_forge`,
  `weaver_resolved_schema`, `weaver_resolver`, `weaver_semconv` workspace
  deps. No new external dependencies beyond what the workspace already pins.

## Phases & Todos

### Phase 1: Foundation Runtime Library (`crates/metrics-sdk/`)

Hand-written types that generated code targets. All live in the new
`crates/metrics-sdk/` crate. No modifications to existing crates.

#### 1a: Dimension trait and implementations

**File**: `crates/metrics-sdk/src/dimension.rs`

```rust
pub trait Dimension: Copy + 'static {
    const CARDINALITY: usize;
    const ATTRIBUTE_KEY: &'static str;
    const ATTRIBUTE_VALUES: &'static [&'static str];
    fn index(self) -> usize;
}
```

Implement for `SignalType` (Logs/Metrics/Traces → 3) and
`RequestOutcome` (Success/Failure/Refused → 3). These impls live
in `crates/metrics-sdk/` (re-import the enum types from their
home crates via dependencies).

#### 1b: Precomputed batch types

**File**: `crates/metrics-sdk/src/precomputed.rs`

Types for holding precomputed Arrow RecordBatches:

```rust
/// Holds the precomputed metrics and attributes record batches
/// for a single metric set at a specific MetricLevel.
pub struct PrecomputedMetricSchema {
    /// The metrics table (one row per counter). Static after init.
    metrics_batch: RecordBatch,
    /// The attributes table (dimension attrs per data point). Static after init.
    attrs_batch: RecordBatch,
    /// Number of data points per metric (product of active dimension
    /// cardinalities).
    points_per_metric: Vec<usize>,
    /// Total number of data points across all metrics.
    total_points: usize,
}
```

Builder interface for the NumberDataPoints table:

```rust
pub struct CounterDataPointsBuilder {
    parent_ids: Vec<u16>,      // precomputed: which metric each point belongs to
    start_time: i64,           // set once per collection interval
    // Built at runtime:
    time_unix_nano: Vec<i64>,
    int_values: Vec<Option<i64>>,
    double_values: Vec<Option<f64>>,
}

impl CounterDataPointsBuilder {
    /// Create from a PrecomputedMetricSchema.
    pub fn new(schema: &PrecomputedMetricSchema) -> Self;

    /// Set all data point values from a counter array snapshot.
    /// The snapshot is a flat array matching the schema's point layout.
    pub fn set_values(&mut self, timestamp: i64, values: &[u64]);

    /// Build the NumberDataPoints RecordBatch.
    pub fn finish(&mut self) -> Result<RecordBatch, ArrowError>;
}
```

#### 1c: OtapPdata assembly

**File**: `crates/metrics-sdk/src/assembly.rs`

Helper to combine the 3 tables + resource/scope info into an
`OtapArrowRecords::Metrics(...)` payload, producing `OtapPdata`.

### Phase 2: Schema Format & Weaver Integration

#### 2a: Schema YAML design

Define the SemConv YAML format with `x-otap-levels` extensions for
the consumer/producer pilot:

```yaml
groups:
  - id: metric.node.consumer
    type: metric
    metric_name: node.consumer.items
    instrument: counter
    unit: "{item}"
    brief: "Items consumed by this node."
    stability: experimental
    attributes:
      - ref: outcome
        requirement_level: required
      - ref: signal_type
        requirement_level:
          conditionally_required: "when metric_level >= normal"
    x-otap-levels:
      basic:
        dimensions: [outcome]
      normal:
        dimensions: [outcome, signal_type]
      detailed:
        dimensions: [outcome, signal_type]
```

File: `crates/telemetry/self_metrics.yaml` (pilot schema)

#### 2b: Weaver template authoring

Create MiniJinja templates that generate:
1. **Measurement struct** — level-aware enum with counter arrays sized
   by dimension cardinality products.
2. **Precomputed schema constructor** — builds the metrics and attributes
   RecordBatches at init time from const data.
3. **Snapshot-to-datapoints encoder** — reads the counter array and fills
   the CounterDataPointsBuilder.
4. **Increment API** — dimension-indexed `add()` methods.

Templates live in: `templates/metrics/` at the workspace root.

#### 2c: xtask generate-metrics command

Add `generate-metrics` to `xtask/src/main.rs`. New module
`xtask/src/generate_metrics/`. It uses the Weaver Rust API directly
(same pattern as `crates/validation/` and
`crates/core-nodes/fake_data_generator/`):

1. Finds all metric schema YAML files in the workspace.
2. Uses `weaver_semconv::RegistryRepo` + `weaver_resolver::SchemaResolver`
   to load and resolve the schema.
3. Converts to `weaver_forge::registry::ResolvedRegistry`.
4. Parses `x-otap-levels` extensions from raw YAML (not in Weaver's
   resolved model).
5. Uses `weaver_forge` to render MiniJinja templates from
   `templates/metrics/`.
6. Writes generated `.rs` files to the crate's `src/` directory.

Add to `xtask/Cargo.toml`:
```toml
weaver_common = { workspace = true }
weaver_forge = { workspace = true }
weaver_resolved_schema = { workspace = true }
weaver_resolver = { workspace = true }
weaver_semconv = { workspace = true }
```

#### 3c: Integration test

2. Increments various counters with dimension values.
3. Snapshots and encodes to OTAP Arrow RecordBatches.

### Phase 4: Wire Into ITS

#### 4a: Metrics collection path

Create a new collection pathway in `crates/metrics-sdk/` (parallel to
the existing `MetricsReporter` → `MetricsDispatcher` path) that:
1. Snapshots the generated counter struct.
2. Uses `CounterDataPointsBuilder` to encode values.
3. Combines with precomputed metrics + attributes batches.
4. Produces `OtapPdata::Metrics(...)`.

#### 4b: Internal telemetry receiver for metrics

Extend `InternalTelemetryReceiver` (or create a new counterpart) to
accept the `OtapPdata` metrics payloads from the new collection path
and inject them into the pipeline — the same pattern as ITS logs.

### Phase 5: Protometheus ITS Exporter

Replaces the entire current Prometheus stack (`MetricsDispatcher` →
OTel SDK → `opentelemetry-prometheus` → `prometheus` crate). When
complete, the `prometheus` and `opentelemetry-prometheus` crate
dependencies are removed from the workspace, along with
`prometheus_exporter_provider.rs`.

#### 5a: Cumulative accumulator

Arrow-native delta→cumulative accumulation using compute kernels
(or DataFusion). Holds a persistent OTAP batch as the single source
of truth. Schema is stable (precomputed), so accumulation is just
column-wise addition on each delta arrival.

#### 5b: OpenMetrics exposition formatter

Walks the OTAP Arrow batch structure and emits OpenMetrics text.
This is the only Prometheus-aware code in the system.

#### 5c: HTTP `/metrics` endpoint

Clone the cumulative batch on scrape, format, return. Replaces the
existing `prometheus_exporter_provider.rs` axum server.

## Dependency Graph

The SDK must support both delta and cumulative temporality:

- **Delta** is cheaper for internal pipeline transport — just emit the
  diff since last collection. No state accumulation in the SDK.
- **Cumulative** is required for Prometheus-style export.

**Design**: The counter measurement structs are always delta internally
(counters are read-and-reset on each collection tick). The SDK produces
delta `Sum` data points. The precomputed metrics batch encodes
`aggregation_temporality = DELTA` and `is_monotonic = true`.

### End State: Replace All Existing Prometheus Code

When this work is complete, the current Prometheus export path is removed
entirely:

- **Removed**: `MetricsDispatcher` → OTel SDK → `opentelemetry-prometheus`
  → `prometheus` crate → text encoding. The `prometheus` and
  `opentelemetry-prometheus` dependencies are dropped.
- **Removed**: The existing `prometheus_exporter_provider.rs` axum server.
- **Replaced by**: ITS collection → protometheus exporter (the only
  Prometheus-serving component left).

### Protometheus Exporter Architecture

The protometheus exporter is a single component that serves as both
accumulator and HTTP endpoint. It replaces everything in the current
OTel SDK metrics export chain.

**Data flow**:

```
  metric_set structs (delta counters)
       │
       │  collection tick: read-and-reset
       ▼
  OTAP delta NumberDataPoints batch
  (precomputed metrics + attrs tables alongside)
       │
       │  ITS channel
       ▼
  ┌─────────────────────────────────┐
  │   Protometheus Exporter         │
  │                                 │
  │   Persistent cumulative state:  │
  │   One OTAP Arrow batch (or a    │
  │   columnar store equivalent).   │
  │                                 │
  │   On receive(delta):            │
  │     Add delta values into the   │
  │     running cumulative columns  │
  │     using Arrow compute (add    │
  │     kernel or equivalent).      │
  │     Update timestamps.          │
  │                                 │
  │   On GET /metrics:              │
  │     Clone current cumulative    │
  │     batch. Walk the Arrow       │
  │     structures. Emit            │
  │     OpenMetrics text.           │
  └─────────────────────────────────┘
```

**Key architectural properties**:

1. **Single stateful map** — the cumulative batch is the only copy of
   accumulated state. No temporal_reaggregation_processor in the path,
   no double-counting of state.

2. **Arrow-native accumulation** — the delta→cumulative addition uses
   Arrow compute kernels (e.g., `arrow::compute::add` on Int64/Float64
   arrays) or DataFusion if more complex operations are needed. This
   keeps the accumulation vectorized and cache-friendly rather than
   row-by-row map lookups.

3. **Schema stability** — because the metrics and attributes tables are
   precomputed and constant, the cumulative batch schema never changes
   after init. New delta arrivals always have the same shape. The
   accumulator just adds values column-wise.

4. **Scrape isolation** — cloning an Arrow RecordBatch is cheap
   (reference-counted column buffers). A Prometheus scrape gets a
   consistent snapshot without blocking the accumulation path.

5. **OpenMetrics formatter** — walks the OTAP Arrow batch structure
   (metrics rows → data point rows → attribute rows) and emits
   standard OpenMetrics exposition format text. This is the only
   Prometheus-related code in the system.

## Open Decisions

1. **SemConv registry as submodule**: Should we add the OTel
   `semantic-conventions` repo as a git submodule for shared attribute
   definitions (like `outcome`, `signal_type`), or define them inline
   in our schema YAML?
2. **Generated file naming**: `generated.rs`, `metrics_gen.rs`, or
   something else?
3. **Resource/scope**: How to populate resource and scope in the
   precomputed metrics batch — hardcoded pipeline identity, or
   injected at init time?
4. **x-otap-levels in Weaver**: The `x-otap-levels` extension is
   custom and won't be in Weaver's resolved schema. We may need to
   parse it from the raw YAML separately and merge it with the
   resolved registry. Or use Weaver's extension mechanism if it
   exists.
5. **Protometheus HTTP server**: Reuse an existing HTTP server in the
   pipeline (e.g., the health check endpoint), or spin up a dedicated
   one for `/metrics`?
