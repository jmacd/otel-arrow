# Contributing to otel-expohisto

Thanks for your interest! This document covers setup, testing, and project conventions.

## Prerequisites

- **Rust stable** (≥ 1.73) for building and testing
- **Rust nightly** for fuzz testing (`cargo +nightly fuzz`)

## Building and Testing

```bash
# Build
cargo build --features bench-all

# Run all tests
cargo test --features bench-all

# Run clippy
cargo clippy --all-targets --features bench-all -- -D warnings

# Build docs
cargo doc --no-deps --features bench-all

# Run benchmarks
cargo bench --features bench-all

# Run examples
cargo run --example basic
cargo run --example merge
cargo run --example sizing
```

## Fuzz Testing

Four fuzz targets exist for finding edge-case bugs:

```bash
# Install cargo-fuzz (one-time)
cargo install cargo-fuzz

# Run a fuzz target (runs indefinitely — Ctrl+C to stop)
cargo +nightly fuzz run histogram_oracle
cargo +nightly fuzz run merge_oracle
cargo +nightly fuzz run stateful_oracle
cargo +nightly fuzz run rng_stress
```

- **`histogram_oracle`** — Feeds arbitrary values to a histogram and checks
  invariants (count, sum, min, max consistency; bucket integrity)
- **`merge_oracle`** — Tests merge operations between two histograms with
  random configurations
- **`stateful_oracle`** — State-machine fuzzer that interleaves update, merge,
  clear, and read operations in arbitrary order
- **`rng_stress`** — Large-histogram (N=160) stress test with millions of
  random values from configurable distributions

## Architecture

```
src/
├── lib.rs           # Crate root, re-exports
├── histogram/       # Core histogram struct:
│   ├── mod.rs       #   struct, construction, update, downscale, widen
│   ├── merge.rs     #   merge_from
│   ├── view.rs      #   HistogramView, BucketView, BucketsIter
│   ├── quantile.rs  #   quantile estimation
│   ├── downscale.rs #   downscale and widen operations
│   ├── width.rs     #   Width enum and SlotAddr
│   ├── swar.rs      #   SWAR helpers
│   └── tests.rs     #   unit tests
├── mapping.rs       # Scale-to-index mapping dispatch
├── exponent.rs      # Scale ≤ 0 mapping (bit extraction)
├── float64.rs       # IEEE 754 bit manipulation helpers
├── logarithm.rs     # ln()-based mapping algorithm (feature-gated)
└── lookup.rs        # Lookup table mapping (generated by build.rs)

benches/             # Criterion benchmarks
build.rs             # Generates lookup.rs from mapping-gen/
mapping-gen/         # Sub-crate for lookup table generation
fuzz/                # cargo-fuzz targets
examples/            # Runnable examples
```

### Key design decisions

- **Allocation-free**: `Histogram<N>` is entirely stack-allocated. The `N`
  words of data pool hold bucket data; statistics (count, sum, min, max)
  are stored in a separate `Stats` struct field.
- **Sub-byte counters**: Buckets start at 1 bit wide (B1) and widen through
  B2→B4→U8→U16→U32→U64 via SWAR (SIMD Within A Register) bitwise operations.
- **Literal mode**: New histograms store raw f64 values until the pool fills,
  then promote to bucket mode at the optimal scale.
- **Compile-time table generation**: The lookup table and its scale (`scale-1`..`scale-16`)
  are Cargo features — no runtime dispatch overhead.

## Pull Request Checklist

- [ ] `cargo test --features bench-all` passes
- [ ] `cargo clippy --all-targets --features bench-all -- -D warnings` is clean
- [ ] `cargo doc --no-deps --features bench-all` builds without warnings
- [ ] New public API has doc comments with `# Panics` and `# Errors` sections
- [ ] Benchmarks still compile: `cargo bench --no-run --features bench-all`
- [ ] Consider running `cargo +nightly fuzz run stateful_oracle` for a few
  minutes if your change touches histogram internals
