# Auto-sampling for log events: two algorithms

Status: design + reference implementation in
`rust/otap-dataflow/crates/telemetry/src/sampling/`. Not yet wired to
the engine's tracing layer.

## 1. The problem

The OTAP dataflow engine emits internal log events through a
`tracing` subscriber. Under load some callsites fire millions of
times per period while others fire once. We want to:

1. Keep a bounded sample (size `T`, e.g. 100 records per period) so
   memory and downstream traffic stay flat.
2. Skip rejected events **before** their arguments are formatted,
   because formatting is what makes logging expensive in Rust
   (`tracing` already gives us this control via `enabled()`).
3. Keep rare callsites visible alongside chatty ones — a uniform
   reservoir would drown them out.
4. Attach a per-record **sampling weight** so that downstream
   estimators (counts, sums, top-K) remain unbiased after sampling.

This is a streaming sampling problem with two extra wrinkles: the
hot path must be cheap, and the per-record weights must be
chosen so the Horvitz–Thompson estimator (`Σ weight · 1` ≈ raw
event count) is unbiased even though the sampling threshold itself
is being learned online.

## 2. Background

### Priority sampling (Duffield, Lund, Thorup)

Assign each item `i` a uniform random key `u_i ∈ [0, 1)` and a
*sampling weight* `w_i > 0`. Compute its **priority** `k_i = u_i / w_i`,
sort by `k_i`, and keep the `T` items with the smallest priorities.
The threshold `τ` is the `(T+1)`-th smallest priority. The
**Horvitz–Thompson estimator** assigns each kept item the inverse of
its inclusion probability:

```
ŵ_i = max(w_i, 1/τ)
```

`Σ ŵ_i` is an unbiased estimator of `Σ w_i` over all items in the
stream. This is the foundation of both algorithms below.

> Duffield, Lund, Thorup, *Priority sampling for estimation of
> arbitrary subset sums*, J. ACM 54(6), 2007.

### Predictable thresholds (Ting)

The HT proof requires `τ` to be **predictable** with respect to the
σ-algebra generated by `{u_j : j ≠ i}` — informally, `τ` may not
depend on item `i`'s own randomness. Daniel Ting showed that this
holds for adaptive bottom-`k` schemes, which is exactly what we need
when `τ` is learned online from the same stream we're sampling.

> Ting, *Data sketches for disaggregated subset sum and frequent item
> estimation*, arXiv:1708.04970, 2017. Especially §2–3 (priority
> sampling) and the predictable-threshold theorem.

This is the paper the user pointed at to start the design.

### Estimating the unseen (Chao1 + Good–Turing)

Algorithm K (below) needs to assign a sensible weight to a callsite
that did **not** appear in the previous period. The estimator is:

```
unseen_weight = 1 / E[freq of an unseen callsite]
              = 1 / (n · missing_mass / S_unseen)
```

where `S_unseen ≈ Chao1(n, f1, f2) − S_seen` is the estimated number
of unseen callsites and `missing_mass ≈ f1 / n` is the Good–Turing
estimate of the probability mass attached to them.

> Chao, *Nonparametric estimation of the number of classes in a
> population*, Scand. J. Stat. 11, 1984.
>
> Good, *The population frequencies of species and the estimation of
> population parameters*, Biometrika 40, 1953.

Reference Go implementation by the same author:
[`chao1.go`](https://github.com/jmacd/essay/blob/87f081e0fe03998caf66cd950a9d76f824e6d50d/examples/internal/datashape/chao1.go).

## 3. Setting

Both algorithms share an interface and the same operating model:

```rust
pub trait LogSampler<C, P> {
    type Ticket: Copy;
    fn admit(&self, callsite: &C) -> Admission<Self::Ticket>;
    fn insert(&self, callsite: C, ticket: Self::Ticket, payload: P);
    fn flush(&self) -> Vec<(C, P, f64)>;
    fn flush_into(&self, out: &mut Vec<(C, P, f64)>);
}

pub enum Admission<Ticket> { Skip, Admit(Ticket) }
```

- `admit` is the cheap, hot-path test (called from `Layer::enabled()`):
  it draws a random `u`, compares it to a threshold, and returns
  either `Skip` (caller drops the event without formatting) or
  `Admit(ticket)` (caller formats and calls `insert`).
- `insert` adds the formatted record to the in-memory sample.
- `flush` is called at end-of-period (e.g., once per second) and
  returns `(callsite, payload, sampling_weight)` triples.

The sampler is `!Sync`; one instance lives behind a thread-local on
each engine core, so no locking is needed on the hot path.

We need **per-callsite** stratification: the goal is that rare
callsites survive, so we want each item's *weight* to reflect how
many times its callsite has been seen. Both algorithms do this by
deriving the priority-sampling weight from the inverse of the
callsite frequency. They differ in **when** the frequency is
measured.

## 4. Algorithm J — live frequency, buffer + bulk

State: a global threshold `τ` (initially `+∞`), a `freq[c]` counter
per callsite, and a buffer of admitted records of capacity `B > T`.

For each event from callsite `c`:

1. Increment `freq[c]`; let `f = freq[c]`.
2. Draw `u ∈ [0, 1)`.
3. Compute the **frozen** key `k = u · f` (stored on the item; never
   recomputed).
4. If `k ≥ τ`, **skip** (return `Skip`). Otherwise **admit** with
   ticket `(k, f)`.

When `insert` is called with an admitted ticket, push
`Item { key, f_at_admit, callsite, payload }` into the buffer. When
the buffer reaches `B`, run a **bulk sample-down**: partial-sort by
`key`, keep the `T+1` smallest, and tighten `τ` to the new
`(T+1)`-th smallest key.

At flush:

- Run one final bulk sample-down.
- Drop the pivot.
- For each remaining item emit weight `max(1, f_at_admit / τ_final)`.
- Reset state for the next period.

## 5. Algorithm K — lagged frozen weights + Chao1

State: a previous-period frequency table `freq_prev`, a current-
period frequency table `freq_curr`, a single bottom-`T` max-heap, a
threshold `τ`, and a scalar `unseen_weight`.

For each event from callsite `c`:

1. Increment `freq_curr[c]`.
2. Look up `w_c`:
   - if `c ∈ freq_prev`: `w_c = 1 / freq_prev[c]`,
   - else: `w_c = unseen_weight`.
3. Draw `u ∈ [0, 1)`, compute `k = u / w_c`.
4. If `k ≥ τ`, **skip**; else **admit** with ticket `k`.

`insert` pushes `(k, callsite, payload)` into the heap. If the heap
exceeds `T`, pop the max and update `τ` to the new heap root.

At flush:

- Drain the heap; emit each item with weight
  `max(1, 1/(τ_final · w_c))`.
- If `Σ freq_curr ≥ min_period_count` (default 32) the period is
  long enough to be a useful reference: swap `freq_prev` and
  `freq_curr` and set `unseen_weight = chao1_unseen_weight(freq_curr)`.
  Otherwise discard `freq_prev` and reset to warmup.
- Reset `τ`, clear `freq_curr`.

### Why this works

The weights are **frozen** for the entire period (chosen at the
start, from prior data), so the heap's threshold `τ` is monotone
and the skip test is trivially conservative. This is just standard
streaming priority sampling — the only novelty is the choice of
weights. The Chao1/Good–Turing imputation gives unseen callsites a
principled weight rather than guessing.

The trade-off vs. J: K reacts to changes in callsite frequency one
period late. In a stationary regime that's fine; in a regime shift
the first post-shift period is somewhat off until the new frequencies
freeze in.

## 6. Comparison

| Property                              | J (R1 frozen)                   | K (lagged + Chao1)             |
| ------------------------------------- | ------------------------------- | ------------------------------ |
| Buffer beyond `T`                     | Yes (`B > T`)                   | No                             |
| Per-event hot-path data structure     | HashMap (freq) + Vec push       | HashMap (freq) + heap push/pop |
| Reacts to frequency changes within…   | …the same period                | …the next period               |
| Needs warmup?                         | No                              | Yes (one period)               |
| Unseen callsites                      | Always admitted (`f=1` ⇒ `k=u`) | Imputed via Chao1              |
| Per-callsite expected admits          | `τ·(1+ln(N_c/τ))`               | ≈ T proportional to `freq`     |
| Skip-test correctness depends on…     | …per-item frozen key            | …weights frozen for the period |
| `Σ weight` unbiased                   | Yes (verified by Monte Carlo)   | Yes (verified by Monte Carlo)  |
| Per-event cost (measured, see §7)     | ~8 ns                           | ~16 ns                         |

Both algorithms are correct in the HT sense. The choice between
them is essentially: **J trades higher buffer churn for in-period
adaptation; K trades a one-period delay for a smaller, simpler hot
path with no buffer.**

## 7. Benchmarks

Hardware: single thread, release build with debuginfo, on the
author's WSL2 machine. Workload is a Zipf-like callsite distribution
over 64 callsites; payload is a `u64`. Throughput is total events
processed (admit + maybe insert + final flush).

```
mode      ns/event   M events / s
-------------------------------------
baseline       0.10       9 585.3   (just iterate the slice)
rng            0.95       1 057.5   (SmallRng + one f64 draw per event)
hashmap        6.68         149.6   (rng + per-event HashMap entry++)
J              8.26         121.1   (full Algorithm J)
K             16.40          61.0   (full Algorithm K, warmed up)
```

Read top-down: the **per-event cost of J is dominated by the
HashMap entry update** (~80% of J's ~8 ns budget). The RNG is ~12%.
The actual sampling logic (multiply, compare, optional Vec push,
amortized partial sort) adds a tail of about 1.5 ns. K pays the same
HashMap cost plus an additional ~8 ns for the heap push/pop and
divide.

Criterion's batch-level numbers (per 10 000 events) corroborate this:
J runs in ~95–105 µs and K in ~200–220 µs across uniform, Zipf-like,
and bursty workloads, with K consistently ~2× more expensive than J.

> Reproduce: `cargo bench -p otap-df-telemetry --bench sampling`
> for the criterion runs, or
> `cargo run --release -p otap-df-telemetry --example profile_sampling -- <mode>`
> for the per-mode breakdown above.

## 8. Allocations

The hot path allocates **nothing** in steady state. All collections
(freq tables, buffer, heap) are sized once and reuse capacity across
periods (`HashMap::clear` and `Vec::drain` keep the backing storage).

Per period (steady state), each `flush` performs **one** allocation:
the returned `Vec<(C, P, f64)>`. To eliminate even that, callers may
use `LogSampler::flush_into(&mut Vec)` and recycle a single output
buffer across periods.

The first period is warmup: the freq HashMap grows from empty to
roughly `K_distinct` entries. A future `with_capacity_hint(target,
expected_callsites)` constructor would let callers pay this cost up
front if they know the cardinality.

## 9. What's next (open work)

1. **Plug in `tracing::callsite::Identifier`** as the production `C`
   type. This is `Copy` and equality is pointer-equality, so a
   custom `BuildHasher` (e.g. an identity hasher) would skip
   SipHash entirely. Profiling above suggests this could roughly
   halve J's per-event cost.
2. **Wire to the engine.** A `tracing::Layer` adapter calls
   `admit()` from `enabled()` (so skipped events never format) and
   `insert()` from `on_event()`. A periodic task on the engine core
   calls `flush_into()` and forwards the batch through the existing
   `flume::Sender<ObservedEvent>` as a new `LogBatch` variant, so
   the internal-telemetry receiver builds a single OTLP request per
   period.
3. **Pick J or K** for production. J is faster, adapts within a
   period, and has no warmup; K is simpler, has no buffer, and
   handles unseen callsites with a principled weight. The author
   leans toward J for the engine telemetry use case where
   per-period regimes change frequently.
